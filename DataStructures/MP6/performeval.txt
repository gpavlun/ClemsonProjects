below, but to summarize;
My code performed close to the formula across the board, but the trend breaks down a
bit at either end of the spectrum. My code outperformed the formulas at small load factors,
but at higher load factors it began to underperform. This is possibly due to the computer
not being able to generate true random numbers, or it could be that the formula uses the
load factor as a constant, when it is more likely to be an exponential.

Random addressing for retrieve driver with seperate chaining. The supporting data is shown 
below, but to summarize;
This was by far the best performing algorithm, despite conceptually not seeming to be.
I would have assumed initailly that the linear search at each node would produce bad
search times, but in general it worked very effectively. Not only that, it was also
much easier to implement, in my opinion, and had far less edge cases to worry about.
The code performed worse in all cases with unsuccessful searches, but performed better
in all successful cases. This may be due to my own failure to implement the probing
correctly, but I would argue that this is due to my implementation that keeps all the
nodes initialized and cleans each linked list to never have empty spaces after a 
deletion.

This section will analyze primary clustering of each method with random v.s. sequencial 
addresses. As always, data is list below, but a summary is provided here;
The performance for seperate chaining is pretty much the same across the board, no notable
difference with it, which is to be expected given it isn't really succeptable to random
values. The double hashing protocol had a slight dip in performance from random to sequential,
but overall seemed fairly similar. Linear probing, however, was absolutely nuked by this.
The average probes went from 5 to 4000 in the case of successful searchs. Absolutely terrible.
