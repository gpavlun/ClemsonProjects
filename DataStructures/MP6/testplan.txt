**This section is about double hashing, which will be very similar to linear
probing, but the deincrement will be different. In this case, the table 
will deincrement by a hashed value from the initial starting point, instead
of by one. This may lead to an event where the table is not able to visit
every position in the table, in that case, it will terminate that attempt
when it detects and infinite loop. This is determined if the current position
ever loops back to itself, as this would mean that the deincrement in a multiple
of the table size.

**This last section is with regards to seperate chaining. The table created
is a 10 element table which will never be returned as full. The elements
are chained together at each index, essentially creating a linked list at
each position. The table does not have as many edge cases or issues due to
its inability to be full and not having to deal with odd increments. This
does, however, result in some slower search times for poor hashfunctions.

**Notes on weird occurances. If the table increment loops
through the entire table without hitting every position,
it may result that is unable to find a spot, which will
then result in an infinite loop. The situation in which
this happens is below





building table of type 1 with size 4
increment 4
attempt to insert item in table
	key k inserted, starting at 3, increment 4	
Table[0] = EMPTY
Table[1] = EMPTY
Table[2] = EMPTY
Table[3] = k
increment 4						**as we can see, the increment is the same size as the table
attempt to insert item in table	  which means it will never actually go anywhere. In this case
	key c did not insert		  the program will simply stop trying. This was done with a custom
Table[0] = EMPTY				  double hash function that just returns the size of the table, so
